# 夸克分享转存与媒体扫描联动设计

本设计让夸克分享转存、扫描范围保存和媒体库刷新形成一次完整操作，并记录另一台测试机器的 TMDB 连接失败结论。

## 背景

看影音已经具备夸克分享内容检查、转存任务轮询、转存历史记录和来源扫描能力，但现有链路存在两个断点：

1. 分享导入页只显示来源中预先保存的默认转存目录。目录未设置时，页面显示“未设置”并禁用转存按钮，用户无法在当前页面完成配置。
2. 转存成功后虽然会触发来源扫描，但扫描器只遍历媒体根目录。默认转存目录与媒体根目录相互独立，目标目录不在扫描范围内时，新视频不会进入媒体库。

本次迭代将两个断点合并处理，使用户能在分享导入页选择目标目录，并保证该目录中的转存资源进入后续扫描范围。

## 目标

- 在夸克分享导入页直接选择或更改转存目录。
- 保存默认转存目录，并在需要时自动将其加入媒体根目录。
- 支持一次选择多个分享条目，并在本批转存结束后只执行一次来源扫描。
- 扫描结束后刷新网盘资源页和本地聚合媒体库，使新视频无需重启即可出现。
- 对“转存失败”和“转存成功但扫描失败”给出不同反馈。
- 保持用户原始网盘文件不变，不执行移动、重命名或删除。

## 非目标

- 不增加公共在线资源搜索或非用户自有媒体入口。
- 不绕过夸克登录、分享提取码或服务端权限校验。
- 不引入非官方 TMDB 地址，不关闭 TLS 证书校验。
- 不修改播放器控件层级、动画或播放行为。

## 方案比较

### 方案一：导入页选择目录并持久化扫描范围

在导入页复用夸克目录选择器。选中的目录保存为默认转存目录；若它不在现有媒体根目录之下，则把该目录追加为媒体根目录。转存成功后执行一次统一刷新。

优点是操作闭环完整，后续全量扫描也能持续发现转存资源。缺点是选择任意新目录会扩大来源的扫描范围，但这是用户已确认接受的行为。

### 方案二：要求用户预先在来源设置中配置

保留当前数据结构，只在页面提示用户返回设置。实现改动较少，但操作被拆成多个页面，仍无法解决截图中的无入口状态。

### 方案三：临时扫描转存目录

转存完成后临时扫描目标目录，不修改媒体根目录。首次刷新可以发现资源，但后续全量扫描可能移除或遗漏该目录，结果不稳定。

采用方案一。

## 交互设计

导入页顶部的“转存到”区域显示当前目标路径，并提供“选择目录”或“更改目录”按钮。未配置时不再只显示不可操作的“未设置”。目录选择器使用现有 `QuarkDirectoryPickerPage`，保持来源编辑页的一致操作方式。

选择目录后立即更新当前页面状态。分享内容仍通过“查看分享内容”读取，用户勾选一个或多个文件夹或文件后执行“转存所选内容”。目录配置未成功保存时，不发起转存请求。

转存和扫描期间禁用目录选择、分享检查、勾选和转存按钮，并使用现有按钮位置展示进行中状态，不改变页面整体层级和动效。

成功反馈分为两类：

- 转存与扫描均完成：提示“转存完成，已扫描到媒体库”。
- 转存完成但扫描或索引刷新失败：提示“文件已转存，但媒体库刷新失败，请重试扫描”。

## 数据与目录范围

页面以仓库中最新的 `CloudSource` 为准，不长期依赖路由传入的旧快照。目标目录选择完成后生成更新后的来源：

- `defaultTransferDirectory` 设置为所选 `CloudRemoteRef`。
- 若 `CloudSourcePathScope.containsSourcePath` 已覆盖该目录，则保持媒体根目录不变。
- 若未覆盖，则将该 `CloudRemoteRef` 追加到 `rootRefs`，并同步生成 `rootPaths`。
- 保存时保留来源名称、启用状态、扫描统计和已有凭据；Cookie 不读取到页面，也不写入日志。

路径覆盖继续使用标准化后的严格目录边界。例如 `/影视` 覆盖 `/影视/电影`，但不覆盖 `/影视库`。已被上级媒体根目录覆盖的转存目录不会重复加入。

## 转存事务

`QuarkImportController` 增加批量导入入口，接收同一分享中的多个选中条目：

1. 在历史仓库的同一把锁内预检全部条目；全部可转存时，一次写入本批待处理记录。
2. 使用服务现有的批量参数发起一次保存任务。
3. 将同一任务编号写入本批记录并轮询到终态。
4. 服务端成功后，将本批记录标记为成功。
5. 调用一次来源刷新协调器，不按条目重复扫描。

如果任一条目已经处于转存中或已成功转存，则在提交服务端任务前拒绝本批操作，且不为其他条目留下待处理记录。服务端任务失败、超时或取消时，更新本批记录状态且不触发扫描。

扫描失败发生在服务端转存成功之后，因此不得把转存历史回写为失败。控制器返回可区分的结果，让页面显示部分成功提示和手动重试建议。

## 刷新流程

转存成功后复用 `CloudSourceRootRefreshCoordinator.refreshSource`，按既有顺序完成：

1. 重新读取当前本地聚合索引和网盘资源快照。
2. 扫描更新后的夸克来源。
3. 再次刷新本地聚合索引和网盘资源快照。

这样可保证扫描器读取已经持久化的新媒体根目录，并让两个媒体库界面在扫描结束后同步看到新视频。

## TMDB 日志结论

附件日志中的 TMDB 失败发生在连接阶段：对官方 API 地址的直连探测连续在 5 秒后超时，之后常见本机代理探测也没有找到可用路径。日志没有出现 HTTP `401` 或 `403`，说明请求没有进入 API Key 鉴权阶段。

另一台测试机器使用与本机相同且本机可用的 API Key，因此可以排除密钥内容差异。根因范围收敛为另一台机器的 DNS 解析、网络出口、代理或防火墙环境。应用继续保持 TMDB 不可用时不阻断本地扫描、网盘扫描和播放。本次不通过替换非官方域名、绕过证书校验或内置代理来掩盖外部网络问题。

## 错误处理

- Cookie 缺失：保持现有提示，要求先编辑来源。
- 分享链接或提取码无效：不改变目录配置和已有媒体索引。
- 目录保存失败：不发起转存，保留当前选择并提示重试。
- 重复转存：在服务端任务创建前拒绝整批操作。
- 转存任务失败、超时或取消：记录对应状态，不执行扫描。
- 转存成功但扫描失败：保留成功历史，提示用户手动重新扫描来源。
- 页面销毁：释放自建服务和监听器，不删除网盘文件或本地索引。

## 测试策略

按测试驱动方式覆盖以下行为：

- 路径范围判断能区分已覆盖目录、相邻前缀目录和重复目录。
- 未设置目标时，导入页提供可操作的目录选择入口。
- 选择未覆盖目录后，来源同时保存默认转存目录和新增媒体根目录。
- 选择已被上级根目录覆盖的目标时，不创建重复根目录。
- 多个选中条目只创建一次转存任务，并只扫描、刷新一次。
- 任一条目重复时，整批请求不会调用服务端保存接口。
- 转存失败不扫描；转存成功但扫描失败仍保留成功记录。
- 页面分别展示完整成功和部分成功提示。

完成定向测试后运行完整 `flutter test`、`flutter analyze` 和 Windows Release 构建。
