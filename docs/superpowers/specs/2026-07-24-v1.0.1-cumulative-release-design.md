# 看影音 1.0.1 累积正式版发布设计

## 目标

将当前 `2.1.55` 代码作为看影音 `1.0.1` 正式版发布到 GitHub。新版本必须同步更新应用版本、应用内更新弹窗、发布说明、版本历史、README、测试期望和 MSIX 清单版本，并提供经过签名与哈希校验的 Windows 下载包。

现有 `v1.0` Release 保留；新建 `v1.0.1` Release，设为 Latest。

## 版本与发布渠道

- Dart/Flutter 版本：`1.0.1+10001`
- Windows MSIX 版本：`1.0.1.0`
- 应用内版本：`1.0.1`
- 发布类型：正式版
- Git 标签：`v1.0.1`
- GitHub Release：非草稿、非预发布、设为 Latest

当前已安装版本在发布开始时记录为 `2.1.55.0`。本次不自动卸载或安装应用，也不在面向用户的弹窗、发布说明或 GitHub Release 描述中加入测试版降级安装提示。

## 面向用户的更新说明

`RELEASE_NOTES.md`、`UPDATE_DIALOG_COPY.md` 与 `lib/utils/version_history.dart` 使用同一组核心信息，标题统一为“看影音 1.0.1 正式版”。正文聚焦普通用户能感知的变化：

1. 本地与网盘媒体库支持按目录浏览，夸克、百度和 OpenList 使用统一的目录选择页面。
2. 夸克分享导入可选择转存目录，转存后自动扫描；修复旧转存目录或远程 ID 可能导致漏扫的问题。
3. 本地与网盘海报统一铺满卡片，圆角更平滑，资源操作按钮更紧凑并减少封面遮挡。
4. Anime4K 效率档使用官方快速组合，并仅在画面确实需要放大时启用，降低不必要的性能开销，仅推荐在观看动漫番剧时候使用这个功能。
5. 修复首次桌面快捷方式询问问题；异机安装包在安装前校验哈希、清单身份和证书。
6. 没有 TMDB Key、断网或 TMDB 暂时不可用时，本地与网盘媒体库及播放器仍可使用；更新不会修改或删除原始媒体文件。

版本历史中只保留一个权威的 `1.0.1` 条目。此前未正式发布或早期开发阶段留下的重复 `1.0.1` 记录需要移除，避免应用启动弹窗读取到旧文案。历史 `2.1.x` 测试版记录继续保留在正式版条目之后。

## 代码与测试范围

本次不改变媒体库、播放器或网盘业务行为，只修改发布元数据和相应测试：

- `pubspec.yaml`
- `lib/core/app_version.dart`
- `README.md`
- `RELEASE_NOTES.md`
- `UPDATE_DIALOG_COPY.md`
- `lib/utils/version_history.dart`
- 当前版本与更新弹窗相关测试

测试必须验证 `1.0.1+10001`、`1.0.1.0`、正式版文案、唯一当前版本历史记录以及关键用户功能词保持一致。

## 构建与下载包

使用 `tool/windows/build_signed_release.ps1` 完成 Windows Release 构建、MSIX 封装、签名验证和桌面交付。最终本地交付物为：

- `看影音-1.0.1.msix`
- `看影音-1.0.1-异机安装包.zip`

验证内容包括 Release 可执行文件和 `data/app.so` 为本轮产物、MSIX 身份为 `com.kanyingyin.player`、清单版本为 `1.0.1.0`、签名有效、桌面文件与构建产物 SHA-256 一致。异机安装 ZIP 必须包含固定清单内的 MSIX、公钥证书、哈希文件、安装脚本和说明。

## GitHub Release

最终提交推送到 `main` 后创建标签 `v1.0.1`。先创建草稿 Release 并上传、校验附件，确认无误后再发布并设为 Latest。GitHub 下载附件使用稳定的 ASCII 文件名：

- `KanYingYin-1.0.1.msix`
- `KanYingYin-1.0.1-cross-device-installer.zip`

Release 描述使用与应用弹窗一致的用户变化摘要，并额外说明：普通情况下下载 MSIX；如果新电脑提示证书不受信任，下载异机安装包并按其中说明安装。描述不得宣称公共在线影视搜索或其他不属于项目定位的能力。

## 验证与失败处理

依次通过以下门禁：

1. `flutter test --no-pub`
2. `flutter analyze --no-pub`
3. Windows Release 构建
4. 签名 MSIX 与异机安装包生成
5. 清单、身份、签名、文件清单和 SHA-256 校验
6. Git 状态与关键差异检查
7. GitHub 标签、Release 描述、Latest 状态、附件数量、大小和摘要校验

任一构建或本地校验失败时，不提交发布版本、不创建标签。GitHub 附件未全部上传并校验前不发布草稿 Release；现有 `v1.0` Release 始终保留。
