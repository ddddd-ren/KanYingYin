# Windows 专属清理、网盘配置入口与 Anime4K 自适应设计

日期：2026-07-24
状态：用户已确认

## 背景

看影音首版只支持 Windows，产品范围是本地与个人网盘媒体库、TMDB 元数据和播放器。当前工程已经移除 Android、iOS、Linux、macOS 和 Web 平台目录，但 Dart 代码、依赖、资产和配置中仍有移动端及旧项目残留。例如外观页仍显示动态配色，播放设置仍保留 Android 渲染器、OpenSLES 和自动画中画分支，`pubspec.yaml` 仍声明多平台插件及图标配置。

旧代码还沿用 `BangumiItem`、`bangumiId`、`matchWithBangumi` 等命名。现有行为已转为本地或网盘媒体播放和 TMDB 刮削，但这些命名增加误解，也可能掩盖已经失效的在线能力。另一方面，旧索引和 Hive 数据可能仍使用这些字段及类型编号，不能直接破坏读取兼容。

网盘媒体库当前的空状态推荐 OpenList 和夸克，没有直接提供百度入口；已有来源后的工具栏也缺少常驻的快速添加入口。这与夸克、百度作为主要个人网盘入口的产品定位不一致。

## 目标

1. 将运行时代码、设置、依赖、资产和构建配置收敛为 Windows 专属实现。
2. 删除公共在线影视搜索、插件规则、WebView 解析、在线评论和旧项目品牌残留。
3. 将仍在使用的旧项目模型和方法改为本地媒体、网盘媒体或 TMDB 语义。
4. 保留旧索引和 Hive 数据的最小只读兼容，避免用户媒体资料突然消失。
5. 在网盘媒体库的空状态和已有来源工具栏中直接提供夸克、百度配置入口。
6. 减少 Windows MSIX 中无关插件和资产，同时保持现有 Windows 功能与交互行为。
7. 优化 Anime4K 的启用条件、失败恢复、运行状态反馈和自动化测试，避免无效 GPU 负载。

## 非目标

- 不重构本地媒体识别、网盘扫描、TMDB 匹配或播放器架构。
- 不删除 OpenList；它继续作为个人媒体入口保留在来源管理页，并明确标注调试中。
- 不修改或删除本地原始视频、网盘文件、网盘目录或远程 ID。
- 不为了清理命名而改变播放器控件层级、动画时长、动画曲线或交互手势。

## 总体方案

采用分层清理，按以下顺序推进：

1. 建立 Windows 功能保留清单和残留清单。
2. 完成网盘配置入口与设置页可见内容清理。
3. 删除非 Windows 运行分支、工具类和存储键。
4. 将活动模型和方法从旧项目命名迁移到中性媒体或 TMDB 命名。
5. 删除不再可达的依赖、资产和多平台配置。
6. 实现 Anime4K 自适应策略、可等待执行和运行状态反馈。
7. 重新生成 Windows 插件注册和代码生成文件。
8. 执行完整测试、静态分析、Windows Release 和签名 MSIX 验证。

每一层完成后先通过定向测试和静态分析，再进入下一层。不能把所有删除集中到一次不可诊断的大改动中。

## Windows 功能保留边界

以下能力必须保留：

- 本地文件夹媒体库、递归扫描、媒体识别和海报墙。
- OpenList、夸克和百度个人网盘来源、扫描、字幕和播放。
- TMDB 自动刮削、手动匹配、海报、背景图和季集资料。
- Windows 窗口、托盘、快捷方式、系统标题栏和文件选择。
- OLED 纯黑背景、系统字体、深浅主题和手动配色方案。
- 硬件解码、Windows 视频渲染、Anime4K、字幕、音轨、选集、全屏、画中画、截图和外部播放器。
- Windows 安全存储、代理、脱敏日志和诊断导出。

候选删除项必须先证明没有 Windows 有效调用路径。名称包含 Android、iOS 或 Linux 不是唯一删除依据；若代码承载 Windows 行为，先提取 Windows 实现再删除平台残留。

## 网盘配置入口

### 空状态

网盘媒体库没有可用来源时显示两个主要操作：

- 添加夸克网盘
- 添加百度网盘

空状态不再推荐 OpenList。OpenList 仍可从“管理网盘来源”进入。

### 已有来源

现有工具栏保留来源切换、刷新、整理、刮削和管理入口，并新增常驻“添加网盘”菜单。菜单只提供：

- 添加夸克网盘
- 添加百度网盘

添加页面返回后，网盘媒体库重新加载来源；若新增来源存在且可用，优先选择新增来源，否则保持原选择。刷新失败时保留原媒体索引并提示用户重试。

### 路由与测试注入

`CloudResourcesPage` 增加百度回调，并复用现有设置模块路由：

- `/settings/cloud-sources/quark/edit`
- `/settings/cloud-sources/baidu/edit`

组件测试通过注入回调验证入口，不依赖真实凭据或网络。

## 设置与运行时代码清理

### 外观设置

- 删除动态配色开关、说明、Provider 状态、存储键和 `dynamic_color` 依赖。
- 保留深色模式、手动配色、系统字体、OLED 优化和系统标题栏。
- 外观页说明改为 Windows 实际支持的主题、字体和桌面显示能力。

### 播放设置

- 删除 Android 视频渲染器页面及路由。
- 删除 OpenSLES、Android 自动画中画、移动端后台行为和对应存储键。
- 删除 Windows 永远不会执行的移动端音量、亮度、截图和渲染分支；如果相关函数同时服务 Windows，先拆出 Windows 实现。
- 保留 Windows 硬件解码、低内存模式、后台播放、自动跳转、自动连播、字幕、播放器动画、错误提示和诊断工具。

### 平台工具与依赖

重点审计并在确认 Windows 无调用后删除：

- `dynamic_color`
- `flutter_displaymode`
- `saver_gallery`
- `screen_brightness_android`、`screen_brightness_ios`、`screen_brightness_ohos`
- `audio_service_mpris`
- 非 Windows 的 media-kit 依赖覆盖
- 仅移动端使用的图标、启动图配置和资源
- 不再使用的 `cupertino_icons`、移动端画中画和平台工具

`flutter_volume_controller`、`audio_session` 等依赖需根据 Windows 播放和音频焦点调用结果决定，不能仅因存在移动端分支就删除。

依赖变更后使用 Flutter 工具重新生成 Windows 插件注册，不手工维护失效注册项。

## Anime4K 自适应优化

### 状态模型

播放器分别维护“用户选择档位”和“当前运行状态”。用户选择档位继续使用现有三档：

- 关闭
- 效率档
- 质量档

当前运行状态使用强类型表示：

- 已关闭
- 正在等待视频或输出尺寸
- 已选择但当前无需启用
- 正在加载
- 效率档已启用
- 质量档已启用
- 加载失败后已关闭
- 当前渲染器不兼容

设置页保存的是用户选择档位，不把窗口缩小导致的临时关闭写回设置。播放器菜单在不改变现有层级和动画的前提下显示当前运行状态，避免用户把“已经选择”误认为“正在消耗 GPU”。

### 自适应判定

策略实现为不依赖 mpv 和 Flutter 控件的纯逻辑组件，输入包括：

- 用户选择档位
- 视频源宽高
- 播放器显示区域宽高和设备像素比例
- 当前画面适配方式
- 渲染器是否支持 GLSL 着色器

播放器显示区域由现有播放表面的布局约束换算为物理像素；视频源尺寸来自播放器宽高流。策略按当前画面适配方式计算实际缩放倍率。只有输出尺寸相对源画面放大超过 5% 时才启用 Anime4K，避免窗口微小变化造成反复切换。

尺寸尚未就绪时不提前加载着色器，显示“正在等待画面尺寸”。原生分辨率已经覆盖输出区域、窗口缩小或画面只被缩小时，清空 Anime4K 着色器并显示“已选择，但当前未启用”。

窗口、全屏和画面比例变化使用 250 毫秒防抖。只有自适应判定结果发生变化时才调用 mpv，不因连续布局事件重复编译相同着色器。

### 执行与失败恢复

将 mpv 着色器命令封装为可注入、可测试的执行器。执行流程为：

1. 标记正在加载。
2. 等待播放器和视频控制器初始化。
3. 设置效率档或质量档着色器列表，或清空着色器。
4. 命令成功后才更新当前运行状态。
5. 记录用户档位、实际状态、视频尺寸、输出尺寸、着色器数量和耗时；日志不记录媒体路径。

所有界面调用必须 `await` 切换结果。加载失败时立即尝试清空 `glsl-shaders`，将实际状态改为关闭，继续播放并只提示一次“当前显卡或渲染器无法启用超分辨率”。失败不会自动尝试效率档，也不会保留错误的已启用高亮。

播放器初始化时应用默认档位也走同一执行器。Anime4K 失败不能让视频初始化失败；本地和网盘视频继续打开。

### 着色器组合

本阶段不修改 Anime4K GLSL 文件内容，也不在用户选择质量档后擅自降级为效率档。性能收益来自：

- 无放大需求时不加载着色器。
- 只在判定结果变化时编译。
- 效率档继续使用轻量组合，质量档继续使用完整组合。
- 关闭或失败后明确清空 mpv 着色器列表。

后续若要根据实时掉帧自动降级，应另行设计，不能在本阶段静默改变用户档位。

## 旧项目命名与数据兼容

### 当前模型

活动代码改用中性语义，例如：

- `BangumiItem` 改为播放媒体条目模型。
- `matchWithBangumi` 改为 TMDB 批量刮削语义。
- `isMatchingBangumi` 和进度字段改为 TMDB 匹配语义。
- 旧 Bangumi 请求头、代理探测和在线能力在确认不可达后删除。

新写入的本地索引不得再包含 `bangumiId`、`bangumiName`、`bangumiSummary` 等旧字段。

### JSON 兼容

旧字段只允许出现在隔离的兼容解析器中。解析器读取旧 JSON 后构造当前媒体/TMDB 模型，当前模型的 `toJson` 不再写回旧字段。旧字段转换失败时保留视频索引，丢弃无效旧元数据，并允许用户重新刮削。

### Hive 兼容

保留现有 Hive `typeId` 的读取能力。活动模型可以改名，但适配器必须继续读取旧字段编号。不得复用已经发布的 `typeId` 表示不兼容类型。

若现有 Hive 条目仍被使用，首次读取后通过仓库原子写入当前结构；迁移完成前不能删除兼容适配器。兼容层不得发起 Bangumi 网络请求，也不得出现在界面文案中。

## 错误处理与安全

- 删除或迁移配置时，不清空用户媒体来源、索引、播放历史或凭据。
- 旧数据解析失败只影响对应元数据，不影响本地扫描和播放。
- TMDB 无 Key、断网或不可用时，本地与网盘扫描、媒体库和播放器继续工作。
- 网盘添加入口只创建配置流程，不自动修改、移动或删除网盘文件。
- 依赖清理导致 Windows 插件缺失时，构建必须失败而不是在运行时静默跳过。

## 测试策略

### 先写失败测试

1. 网盘空状态只显示夸克和百度入口。
2. 已有来源工具栏始终提供夸克和百度添加菜单。
3. 添加完成后重新加载并选择新增来源。
4. 外观设置不再包含动态配色或移动端说明。
5. 播放设置不再包含 Android 专用设置和路由。
6. 当前本地索引序列化不再写入旧 Bangumi 字段。
7. 旧 JSON 和 Hive 数据仍能迁移为当前模型。
8. Windows 插件注册和 `pubspec.yaml` 不再包含已删除依赖。
9. 源码中不再出现可执行的公共在线影视、WebView 解析、规则插件或在线评论能力。
10. Anime4K 只在输出画面需要放大时启用，尺寸变化不会重复执行相同命令。
11. Anime4K 加载失败后清空着色器、继续播放并显示关闭状态。
12. 用户选择档位与当前实际状态在设置页和播放器菜单中保持一致语义。

### 完整验证

- `flutter test --no-pub`
- `flutter analyze --no-pub`
- `flutter build windows --release --no-pub`
- 生成签名 MSIX
- 验证 MSIX 清单身份、版本、架构和签名
- 检查 Release 与 MSIX 中的插件和资产清单
- 将最终安装包复制到桌面

## 交付与提交

清理完成后递增应用版本和 MSIX 版本，更新 `RELEASE_NOTES.md`、`UPDATE_DIALOG_COPY.md`、`README.md` 和 `lib/utils/version_history.dart`。发布文案面向普通用户，说明设置已收敛为 Windows、网盘配置入口更直接、旧数据可继续使用，并明确不会删除原始媒体。

提交只包含本轮源码、测试、依赖、生成注册文件和版本资料，不提交构建目录或安装包。

## 实施分组

同一版本包含三个独立验证的实施组：

1. Windows 专属清理与旧数据兼容迁移。
2. 网盘媒体库夸克、百度配置入口。
3. Anime4K 自适应、失败恢复和状态反馈。

每组先完成失败测试、实现和定向验证，再进入下一组。最终统一执行完整测试、静态分析、Windows Release 和签名 MSIX 交付。
